## 创意阶段的目标

在创意阶段，创始人的主要目标是以调研为导向的验证：在投入资源构建之前，搜集扎实的证据，证明一个真实的问题确实存在（并且你提出的解决方案能有效解决它）。

实际操作上，创意阶段是一系列创始人必须大致按以下顺序回答的问题：

- —这个问题是否真实、具体、频繁到值得围绕它构建产品？
- —究竟是谁有这个问题，这群人是否构成一个市场？
- —是否已有他人在解决它？如果有，怎么解决的、解决得多好？
- —一个解决方案究竟需要做到什么才能解决这个问题，我的点子做到了吗？

这些追问的结果合起来，回答一个唯一的、终极的问题：这件事值得做吗？

这意味着在行动之前先变得具体。"人们在报销上很头疼"是一个观察。"中型市场公司的财务经理每周要花四个多小时来核对提交的报销，因为他们现有的工具无法与会计软件集成"才是一个可检验的假设。

## 创意阶段的退出标准

创意阶段的退出条件是找到问题—解决方案契合（problem-solution fit）。在你开始构建那个解决问题的东西之前，你已经建立了定性证据——主要来自与真人的真实对话——证明你正在为真实的人解决一个真实的问题。

当你能对以下三点都回答"是"时，你就准备好离开创意阶段了：

1. 1问题是否真实且具体？在此回答"是"，要求你能准确说出谁经历这个问题、他们多久遇到一次、对他们的影响有多严重，以及他们目前是怎么应对的。
2. 2你的解决方案是否针对的是真正的问题？不是你最初假设的那个问题，而是验证过程揭示出来的那个问题。有时它们是一回事，但并不总是。
3. 3你是否有足够的信号支撑动手构建？在这个阶段你永远不会有确定性，一味等待确定性本身就是一种失败模式；但你需要足够的定性证据，让"投入做一个 MVP"成为一个有理有据的决定，而非一次信仰之跃。

## 创意阶段的挑战

创意阶段是你创业旅程中最重要的工作发生的地方，因为这里也是最具决定性后果的错误被犯下的地方：现在弄错某件事，会很快让你萌芽中的事业彻底脱轨。不过，创意阶段的大多数挑战都涉及"前进得比你的理解所能支撑的更快"，因此，深思熟虑、审慎推进的创始人会取得稳定的进展。

### 把"构建"误当成"验证"

挑战：当技术障碍被移除，一位满腔热忱的创始人就有可能跳过创业旅程中最重要的工作：验证他们的点子是否真的是一个人们需要并且会使用的解决方案。

即便在当前智能体编程时代之前，42% 的创业公司失败，是因为他们做出了没人想要的东西。而如今，像 Claude Code 这样的智能体编程方案已经大幅压缩了"我有个点子"和"我有个产品"之间的距离，那个失败率只会继续攀升。

尽管做一个拥有"灵光乍现般绝妙点子"的创始人，从未有过比现在更好的时机，但能如此迅速、如此轻松地搭起一个看起来像产品的原型，反直觉地，也为 AI 原生创业公司带来了一种真正危险的存亡风险。

直到最近，构建还需要真金白银的开发时间和预算，连凑出一个最基础的原型通常都要数月。但如今，技术开发的门槛基本消失了，AI 让创始人太容易直接跳进构建，而不去验证它在真实世界中的实用性。

达成问题—解决方案契合，要求先验证假设、再构建；但许多初次（甚至有经验的）创始人错误地相信，AI 让这一要求可以被跳过，把流程变成了：有个点子 → 立刻做个原型 → 把原型当作验证。原型于是成了"假设一直都对"的理由，却从未真正检验它是否成立。

一个能跑的原型，很容易被误认为是"你在解决一个真实问题"的确凿证据，但它不是。你的原型其实是与潜在用户对话时一个有用的压力测试道具。这些对话本身，才是真正的证据。

### 过早扩张

挑战：当构建毫不费力且即时，你就可能让执行规模远远跑在业务需求之前。

过早扩张，意味着在你真正验证某条产品路径值得投入之前，就已经一头扎了进去。

这一直是创业公司的杀手，但 AI 让创始人极其容易在不知不觉中掉进过早扩张的陷阱。智能体编程助手如此强大，以至于在从未有意识地决定偏离航道的情况下，就很容易把执行规模推到远超"验证问题—解决方案契合"之前。

它会围绕一个根本上有缺陷的前提去生成、测试、调试和重构代码库，所投入的热情，与它面对一个绝妙点子时一模一样。系统里的智慧是你的。这个阶段的最高准则，是让你的"判断与理解"始终走在"构建"前面，尤其是当构建如此之快、感觉如此毫不费力的时候。

### 客观性的丧失

挑战：让一个 AI 工具去寻找支持你已有信念的证据，它就会找到。如今，确认偏误配上了一台调研引擎。

确认偏误一直是创业中的职业病：创始人天性就对自己的点子充满热情。如今，AI 工具给确认偏误来了一次大幅强化。让 AI 验证你的创业点子，它会找到支持性证据；让它测算你的潜在市场规模，它会找到那个让你的 TAM 看起来值得投资的数字。

AI 听从你的指挥，这意味着一位不去问尖锐问题的创始人，如今能比以往任何时候都更快地为一个糟糕的点子构建出一个详尽、看起来调研充分的论证，同时还充分自信地以为自己确实在做尽职调查。解药是同一个工具，只是指向相反的方向：AI 会像验证一个点子那样彻底地对它做压力测试。

当调研与结构化的对抗性思考浮现出"你的点子需要修正"的证据时，这就是该转向（pivot）的信号。

## Claude 如何帮助创意阶段的创始人

推动你的 AI 原生创业构想穿越创意阶段，可能感觉慢得没完没了。你是创始人，你就想动手构建。但这个至关重要的启动阶段，本质上是一项调研与验证工作，这意味着要拿起那些能帮你在全力投入写代码之前更严谨地思考的工具。以下是在 Claude 的各个产品形态（[Chat](https://zhida.zhihu.com/search?content_id=275036335&content_type=Article&match_order=1&q=Chat&zd_token=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJpc3MiOiJ6aGlkYV9zZXJ2ZXIiLCJleHAiOjE3ODA2NDYxNDQsInEiOiJDaGF0IiwiemhpZGFfc291cmNlIjoiZW50aXR5IiwiY29udGVudF9pZCI6Mjc1MDM2MzM1LCJjb250ZW50X3R5cGUiOiJBcnRpY2xlIiwibWF0Y2hfb3JkZXIiOjEsInpkX3Rva2VuIjpudWxsfQ.bkxjC3gQkJFze6xYg0XBc9kY1aii7ALw4a-TT6EGLG0&zhida_source=entity)、Claude Cowork 与 Claude Code）中使用它的方法，帮你在做好尽职调查的同时尽快走完创意阶段。

### Chat、Claude Cowork 还是 Claude Code：选择合适的 Claude 形态

AI 让创业创始人更容易更快交付、把繁琐工作流自动化、并以规模化方式运作；而关键在于根据手头任务，决定何时用 Chat、Claude Cowork 还是 Claude Code。

Chat 用于无需离开你正在使用的应用的快速交流。把它用于运营公司时层出不穷的小任务：从一份密集的投资备忘录里提炼一句话要点，董事会前为某个说法做合理性核查，或读懂团队里一条冗长的 Slack 讨论串。

Claude Code 是面向你团队中工程师的智能体编程环境：直接访问代码库、计划模式（Plan Mode）、git 集成，以及本地、IDE 或沙箱化云环境。在这里，一支精简团队在不断壮大的代码库上交付功能，迁移 MVP 时期的遗留代码，并在不等更多人头的情况下从原型走向生产。

三者底层是同一个 Claude；变化的是它周围的工作空间。

## 定义并压力测试问题假设

你自身的领域专长和前期调研已经产生了一个假设。第一项工作是把它打磨到真正可检验。Claude 在这里尤其有用，因为它能强迫你具体化：究竟谁有这个问题，多久一次，多严重，他们目前怎么应对？一个无法精确回答这些问题的问题陈述，还没准备好被验证。

练习：与 Claude 协作打磨你的问题陈述，直到它成为一个可检验的假设。例如，"合同评审耗时太长"并无实质的可检验性。但"中型市场公司的内部法务团队，每个合同评审周期要花 3 天以上，因为修订意见是分散在邮件往来里管理的，而不是一份单一的版本受控文档"就非常可检验。

你的下一步是请 Claude 反驳你的点子，并为你的假设寻找反证。这能浮现出负面市场信号、失败的竞争对手、客户行为模式，以及那些在一份支持性综述里会被悄悄降低优先级的结构性障碍。

目标是在进入客户探索时，你的假设已经针对现有最强的反方论据做过压力测试，从而让信息性的用户访谈真正保持开放式，而不是一场对确认的搜寻。

注意：把 Claude 用作结构化的魔鬼代言人，是 AI 创业生命周期每个阶段的核心用例。

## 市场调研与绘制竞争格局

### 掂量你的竞争对手

有一种创业特有的现象叫"竞争对手忽视"：过度专注于自己的愿景和执行，以至于系统性地低估同一空间里其他人在做什么。所幸，AI 提供了解药：请 Claude 给出最有说服力的论证，说明为什么这个解决方案空间里的某个竞争对手会成功，而你不会。

Claude 能分析他们的做法为什么其实更好、客户为什么会选择他们、你潜在的差异化点为什么可能不如你以为的那么牢固。

练习：请 Claude 按层级绘制你的竞争格局：直接竞争对手、间接竞争对手、潜在收购方，以及可能切入你领域的相邻玩家。然后请它论证为什么每个层级都对你的成功构成真正的威胁，而不只是那个最容易被驳回的版本的威胁。

### 市场调研

Claude Code 能综合公开可得的客户反馈，浮现出反复出现的抱怨和未被满足的需求。额外收获：这样做本质上是对竞争对手客户进行的免费定性调研。

练习：指挥 Claude Cowork 综合你关键来源里的竞争对手评价，找出现有解决方案尚未解决的首要抱怨。如果你的假设解决了其中一个或多个，这就是问题—解决方案契合的有力证据。如果没有，那也值得知道。

Claude Cowork 还能从密集的行业报告、分析师备案文件和市场研究文档中提取相关信息与数据；随后，这些干净、综合后的输入会成为 Claude 分析工作的理想上下文。

练习：基于公开可得的数据构建 TAM/SAM/SOM 模型，并对其背后的假设做压力测试。判断市场是在扩张、整合还是已成熟；这一背景会影响你对时机和差异化的思考。绘制买方格局：谁掌握预算、谁影响决策，以及这两者是否是同一个人。

### 趋势分析

最后，用 Claude 去倾听那些告诉你"是否在正确时机进场"的早期指标。追踪那些已经在讨论你的问题的 subreddit 和 LinkedIn 群组，以及用户描述其困扰时确切使用的措辞。请 Claude 找出曾解决过类似问题的类比市场，并提取哪些奏效、哪些没奏效。浮现出可能加速或威胁这一机会的监管、技术或人口结构趋势。

练习：请 Claude 找出未来两年可能显著影响你市场的三个外部趋势——监管、技术或人口结构——并评估对你特定的假设而言，每一个是顺风还是逆风。

注意：本节的市场调研与竞争格局绘制工作并非一次性任务。你会在 MVP 与上线阶段持续有新发现、持续演进你的思考，因此每当你的假设演变时，重复这些练习都很重要。

## 规划并设计客户探索

你通过与产品潜在用户交谈所学到的东西，其质量取决于：(1) 你提出的问题的质量；(2) 你是否在向正确的人提问。Claude 在开展客户探索方面尤其有帮助，包括：该和谁谈、问什么，以及如何理解你听到的内容。

### 该和谁谈

一份精确的目标画像，远比一长串联系人名单有价值无穷，它包括最可能强烈经历该问题的具体职位、公司类型、团队结构和资历层级。由此出发，找出这些人实际可触达的地方——他们聚集的社区、活动、LinkedIn 群组和 Slack 工作区——并基于"他们离问题有多近"，建立一个"先联系谁"的优先级框架。

### 问什么

目标确定后，用 Claude 来构建访谈框架本身：正确的问题、正确的顺序，结构上设计为浮现"人们实际做了什么"而非"他们以为自己会做什么"。新手创始人的一个错误，是问一个泛泛的、关于未来的开放式问题（"你会用类似这样的东西吗？"），而不是具体地探询相关的过去（"跟我讲讲你上一次处理这个问题是什么时候。"）。

Claude 还能指出你草拟的问题中哪些在诱导受访者、过于宽泛，或者多半会产生噪声而非信号。Claude 也能帮你设计追问，以探查回避、或就重要问题上含糊的回答深挖下去。

如果你的假设涉及不止一个用户画像，Claude 还能为每一个设计不同的问题集。财务经理和 CFO 对同一个问题有着不同的关系，一套单一的访谈框架会抹平这一区别。

练习：先手写草拟你的访谈问题，再请 Claude 审查。具体请它指出任何在诱导、面向未来、过于宽泛，或可能产生"社会期许式"回答而非诚实回答的问题。然后请它为访谈中最可能产生回避的两三个时刻，建议一个追问。

### 访谈后分析

每次对话后，用 Claude 做复盘：把你的笔记喂给它，请它找出哪些确认了你的假设、哪些挑战了它，以及哪些是真正出人意料的。当你收集了一批访谈后，把全套访谈笔记通过 Claude Cowork 跑一遍，浮现反复出现的主题、矛盾，以及两个方向上最强的信号。然后把综合后的输出再带回给 Claude，请它指出你自己对数据的解读，可能在哪里是"模式匹配到你想听的"而非"实际存在的"。

练习：每做完五场访谈，指挥 Claude Cowork 综合你的笔记，产出两份清单：支持你假设的证据，以及挑战它的证据。如果第一份明显长于第二份，问 Claude，这种不对称反映的是数据中实际存在的东西——还是你曾希望找到的东西。

### 客户触达与排期

用 Claude Cowork 自动化围绕建立联系人名单、开展触达、安排用户访谈的运营负担。

Claude Cowork 能用你与 Claude 一起定义的目标画像（包括职位、公司类型和资历层级）去调研并编制一份结构化的潜在对象名单和已核实的联系信息。然后它会规模化地起草个性化的触达邮件，针对每个人的角色和情境量身定制。

随着回复进来，它通过 MCP 连接 Gmail 和 Google Calendar 来管理邮件往来、处理排期请求，把访谈排进日历。工作流持续进行：Claude Cowork 按既定节奏生成跟进草稿（例如对未回复的联系人在第七天跟进），并在每一步完成时更新你的追踪表，让你随时清楚每个潜在对象处在管道中的什么位置。

练习：把你验证过的访谈目标画像交给 Claude Cowork，请它构建一份潜在对象名单、起草一套个性化触达序列，并建立一张追踪表，列出触达状态、跟进节奏和访谈完成情况。然后让它去跑协调工作，而你专注于为对话本身做准备。

## 设计你最终的解决方案构想

你已经做完了验证工作：问题是真实的，你知道谁有这个问题，并且你有一个证据支持的解决方案构想。用 Claude 从各个角度去发展并挑战你的解决方案构想：差距在哪里？存在哪些替代方案？要让这个方案在规模化时行得通，什么必须为真？这是一个重要的现实校验点：这个设计是否真的针对验证过程揭示出的那个问题，而不是你最初进场时假设的那个问题？

练习：把你的解决方案构想呈现给 Claude，请它指出你的设计最依赖的三个假设。然后问，要让每个假设成立，什么必须为真，以及如果其中任何一个不成立，后果是什么。

## 用 Claude Code 构建一个轻量原型

现在到了有趣的部分：有了一个验证过的假设和一个经过压力测试的解决方案构想，你终于准备好动手做点东西了。

这就是创意阶段里 Claude Code 登场的时刻。即便你一路都在小打小闹地折腾，现在才是你产出正式轻量原型的节点：把你的点子摆到一个真人面前、并得到真实反应所需的最小界面面积。

你（暂时）还不是在做一个真实世界的产品；你是在构造一个可用于客户和投资人对话的、你点子的功能样本。真实用户对一个他们真能上手触碰的东西做出的反应，会告诉你十几场问题—解决方案探索访谈都问不出来的东西。之前，你是在确立"你要解决的问题是真实的"；现在，你是在请潜在用户与所提出的解决方案互动。

练习：定义你的解决方案所依赖的那个单一核心交互。指挥 Claude Code 只构建那一个。做出来后，把它摆到你验证过的目标画像中的五个人面前，请他们试用。你在那五场对话中学到的，决定你是继续构建，还是回到绘图板重来。

走到创意阶段的终点，是 AI 创业竞赛中向前的一大跃进，因为现在你不是在押注一个直觉；你是在依据证据执行。接下来是 MVP 阶段，创始人的指导性问题从"这件事值得做吗？"转变为"我们究竟该先做什么？"，而 AI 的主要角色也从调研伙伴转变为施工队。



# 第 4 章 MVP 阶段

很多创始人把 MVP 阶段当作一个施工阶段，但 MVP 阶段本质上仍是一项证据搜集工作。区别在于，你现在搜集的是关于解决方案的证据，而不是关于问题空间的证据；具体来说，是一个真实、可识别的人群是否觉得它足够有价值，以至于会使用它、回头再用、为它付费，和/或把它告诉别人。

## MVP 阶段的目标

作为一家 AI 原生创业公司的创始人，你的目标是把一个验证过的问题转化为一个真实用户真的会用的、可工作的产品。这不是带有路线图上每一项功能的完整版本，而是你点子的最小、最聚焦的一次迭代，它把一个真实的解决方案摆到真实用户面前，并产生产品—市场契合的真实证据。

与此同时，你现在的构建方式决定了以后什么是可能的。这意味着 MVP 阶段还有第二个同样重要的目标：在不积累那种会复利滚雪球的技术债的前提下快速行动——这种技术债，会在真实用户以可观数量到来的那一刻缠住你。

最后，从第一天起就投资于持久的上下文，是让 AI 始终是力量倍增器、而非熵增之源的关键。在一家 AI 原生创业公司里，你的代码库是你与 AI 一次次会话协作的对象，这让"可读性"成为根基。那些跳过规格说明、架构决策和上下文文件（如 CLAUDE.md）的创始人，会撞上一堵可预见的墙：每一次新会话都要重新解释代码库，而 AI 生成的改动会从最初的愿景漂移开来。

## MVP 阶段的退出标准

MVP 阶段的退出条件是产品—市场契合的真实证据：证明一个特定的、可识别的用户群体已经认为该产品足够有价值，以至于会回头再用（留存）、为它付费（营收），或把它告诉别人（推荐）。

## MVP 阶段的挑战

在 MVP 阶段，创始人的最高准则是速度与判断力。这里的挑战集中在：你能否在不走那些日后会让你付出代价的捷径的前提下，以足以见效的速度、用正确的方式、做出正确的东西。

### 智能体式技术债

挑战：因为 AI 实质上移除了过去控制"什么能进入生产"的每一个天然瓶颈，速度有了保证。但当速度是创始人在 MVP 构建中纳入考量的唯一变量时，他们就有可能积累起难以偿还的技术债。

在 MVP 阶段，一些技术债是恰当的，前提是要明白它必须在规模化之前被管理掉。它逐渐累积，可以随时间或在一个专门的冲刺中清掉。然而，AI 技术债会复利滚雪球。

如果没有把规格和架构约束写在 AI 能读到的某个地方，每一次会话都会从头重新推导那些根基性的决策，而这些决策会漂移。你最终得到的代码库背后没有一个连贯的心智模型——不是因为任何单独一块不好，而是因为这些块从未被设计成彼此契合。这是一个真实的问题，而且它往往会很晚才浮出水面。

### 误信虚假的产品—市场契合

挑战：AI 工具能生成亮眼的早期数字，但这些并不保证市场需要你的产品。

早期势头是创始人能拥有的、心理上最强大的体验之一。在数周或数月的验证工作和谨慎、自律的构建之后，交付一个产品感觉就像确认了你一直都是对的。

智能体编程工具能帮你比以往任何时候都更快地到达这一刻，但早期牵引力并不等同于产品—市场契合。上线时的能量来自转瞬即逝的力量，比如你创始人朋友、你投资人旗下其他被投公司的潜在买家，或者一条带来流量峰值的 Hacker News 头条。不幸的是，这些都无法可靠地预测当那股初始助推消退后，第六周或第十二周会发生什么。

### 零摩擦的范围蔓延

挑战：当构建感觉毫不费力且几乎免费时，总会有"再加一个酷功能"或"再处理一个边界情况"。这种范围蔓延弊大于利。

范围蔓延一直是创业风险。如今的区别在于，当加一个功能只要一个下午而不是一个冲刺时，对抗它的传统强制约束力——工程时间的真实成本——不再以同样的方式存在了。

这里的挑战在于，每一项单独的添加都说得过去。产品当然应该处理那个边界情况；用户当然会想要那个工作流。它们在当下并不觉得是范围蔓延，因为用智能体编程构建每一个都花费极少；但随着产品蔓延超出其原始边界，你就有失去方向和势头的风险。

解药是在构建开始之前就写下一份范围定义，描述产品做什么、有意不做什么，以及来自真实用户的、能证明值得新增某项功能的具体证据。这把决策点从"我们该做这个吗？"转变为"已有临界数量的用户告诉我们，没有这个他们就无法从产品中获得价值"。

### 因经验不足而不安全

挑战：创始人用 AI 工具在尚未理解基本安全原则的情况下就把应用赶上市场，最终让用户暴露在本可预防的风险之中。

残酷的真相是，智能体编程工具生成的是能工作的代码，而不是本质上安全的代码。功能性的代码很容易判断，因为功能要么能用、要么不能用。安全漏洞在被利用之前是不可见的，这意味着没有一个天然的反馈回路来提醒一位初次创始人有什么不对劲。然而，把一个上线的 MVP 交付给真实用户，意味着真实的数据、真实的暴露面，以及一旦出错就有真实的后果。

轻视安全对 AI 原生项目来说并非全新现象。各个时代靠自有资金起步的创业公司都常常把安全考量拖到构建很晚的阶段，有时一直拖到生产上线前夕。在任何用户接触你的应用或方案之前先做一次安全评审，是把一个最小可行产品负责任地推向世界的最低门槛。

## Claude 如何帮助 MVP 阶段的创始人

### 在构建之前定义你的架构

在 Claude Code 写下一行生产代码之前，用 Claude 来定义并记录将统御本阶段所有构建的架构决策：要遵循的模式、要避免的依赖、正在做出的取舍及其原因。这份输出会成为一份聚焦的架构上下文文档，并确立 Claude Code 将在其内运作的护栏。

没有这个上下文，每次会话都从头开始，Claude Code 被迫推断它自己的结构性假设。让 Claude Code 在没有护栏的情况下构建，产出的代码库会能工作但结构不连贯，而在不连贯的代码库上迭代和规模化最终是对时间和 token 的浪费。迟早会到一个点，代码不可避免地崩塌，迫使你从头重建。

练习：在打开 Claude Code 之前，先打开 Claude，描述你在构建什么：它解决的核心问题、它服务的用户，以及你对未来六个月现实预期的规模。请它帮你定义应统御你 MVP 构建的架构原则、在你的约束下要避免的依赖，以及你在这个阶段有意识接受的取舍。

接下来，把这份输出保存为 CLAUDE.md markdown 文件。这是你的架构上下文文档：你这次构建的第一个产物，也是此后每一次会话都依赖的那一个。CLAUDE.md 文件充当 Claude Code 的项目级指令，提供项目特定的上下文与指令，当 Agent SDK 在某个目录中运行时会自动读取它们。从功能上说，它们是你项目的持久"记忆"。

### 定义并强制执行你的 MVP 范围

无摩擦的范围蔓延，是 AI 时代 MVP 的标志性失败模式之一。正如你定义并记录了产品的应用架构，你也需要在动手构建任何一个功能之前定义好 MVP 的范围。

Claude 能帮你创建一份范围文档，描述你的 MVP 产品做什么、有意不做什么，以及功能修订标准：在这个节点上，来自真实用户的什么具体证据能证明值得新增某项功能。

当新功能点子浮现时——它们一定会——你用 Claude 来压力测试它究竟是来自用户的真实信号，还是被打扮成产品思考的创始人热情。

### 用 Claude Code 构建你的 MVP

架构和范围一旦定义好，Claude Code 就成为主要的 MVP 构建工具。用它来生成、测试、调试并迭代你的代码库，但把每一次会话都当作对你已经做出的产品决策的执行，而不是抛进一些新决策的机会。

每次 Claude Code 会话开始时，(1) 重温你的范围文档，(2) 把你的 CLAUDE.md 架构上下文文档提供给模型。每次会话结束时，用本次会话浮现出的任何决策来更新它。目标是一个你能解释其结构的代码库，而不只是一个能跑的代码库。

练习：为你的 Claude Code 工作创建一个简单的会话模板，其中包含架构上下文文档、本次会话的具体任务，以及要遵守的任何约束或模式。每次会话结束时，往上下文文档里加一条简短的日志条目，详述构建了什么、做了哪些决策、本次会话引入了什么假设。每次会话五分钟的文档，是抵御"复利滚雪球成不可管理代码库的架构漂移"的廉价保险。

### 在任何用户接触之前做安全评审

作为一家 AI 原生创业公司的创始人，你的责任是知道你的代码库里有什么、理解任何潜在的暴露途径，并且不把明显的漏洞交付给那些把数据托付给你的真实用户。

Claude 能对 AI 生成的代码做一次有用的初轮安全评审，并能帮助识别常见漏洞。把它建进交付前的循环是个好习惯。然而它不能替代安全工具，在更高风险时也不能替代人工评审者——把它当作替代品的创始人，就是最后出现在数据泄露报道里的那些人。

Claude Code Security 更进一步：它扫描代码库以发现安全漏洞，并为人工评审建议有针对性的补丁，浮现出传统方法可能遗漏的问题。

注意：在本电子书出版时，Claude Code Security 是有限测试版发布，因此在把它建进你的工作流之前，请确认当前可用性。

练习：在部署给任何真实用户之前，带着一份具体的简报让 Claude 跑一遍你的核心应用代码：评审认证与会话处理、API 响应中的数据暴露、输入校验与注入风险，以及已知存在漏洞的依赖。认真对待每一项发现，评估它是否需要修复；任何涉及认证、密钥或数据处理的，都要做人工评审。

### 在上线之前构建你的衡量框架

那些把早期牵引力误判为产品—市场契合的创始人，通常正是那些在上线之后才开始追踪数据的人——他们选用的指标是用来评估"什么奏效了"，而不是用来浮现"什么没奏效"。解药是在第一位用户出现之前，就建立你的衡量框架。

用 Claude 来定义对你这个特定产品而言哪些指标重要、基准是什么，以及数据中的什么模式才算真正的产品—市场契合而非令人飘飘然的噪声。具体来说：在发布你的 MVP 之前，设定好你的留存基准、你的激活标准，以及你的第 7 天和第 30 天目标。

接下来，定义对你这个特定产品而言"假阳性"长什么样：例如，注册而不激活、有营收而无留存，或初期热情而无重复使用。当数据到来时，请 Claude 对你自己的牵引力做反方论证：一个怀疑论者会怎么评价这些数字？

### 管理探索与用户反馈的事务

一旦真实用户进入产品，运营层会迅速膨胀。Claude Cowork 处理那些重要但繁琐的工作，比如建立和维护用户联系人名单、开展触达序列、安排反馈会、为缺陷报告分诊，以及追踪迭代周期。在创意阶段管理探索事务的那些 MCP 集成，在这里同样适用。

在收集回路中保留一个人，用于对用户反馈做细致入微的探究。比如一位用户说"这很好，但我希望我还能……"，这需要解读：这是核心需求还是锦上添花？是这个客户特有的，还是代表了一个细分群体？缺失的功能是真正的问题，还是上游入门环节里有别的东西？没有工具能回答这些问题。

练习：配置 Claude Cowork 来运行你 MVP 阶段的反馈回路：向你的早期用户名单起草触达、安排反馈会、为缺陷报告和功能请求设计结构化的接收流程，并就收到的内容写一份每周综述。你自己审阅这份综述；之后，你可以请 Claude 分析这些信息，捕捉你可能忽略的任何重要点。

### 朝证据迭代，而非朝完成度迭代

当你拥有产品—市场契合的真实证据时，MVP 阶段就结束了，无论产品感觉多么"完工"。宣布你已达成产品—市场契合、现在准备从 MVP 阶段进入上线阶段，最终是一项把创始人直觉与已收集证据结合起来的判断工作。不过，有一些有用的试金石：

- —Sean Ellis 测试：问你的活跃用户："如果你再也不能用这个产品，你会有什么感觉？"如果超过 40% 的人会说"非常失望"，那就是一个有意义的 PMF 指标。
- —努力程度测试：在产品—市场契合之前，留存需要持续的人为干预，包括频繁触达、激励、个人跟进，以及创始人为留住用户而付出的英雄式精力。在产品—市场契合之后，产品开始自己完成那项工作。当事情开始从"推"变为"拉"，这一转变是"某种真实的东西已经改变"最清晰的信号之一。

归根结底，没有任何单一数据点能确认产品—市场契合，因为它是一种必须跨多个迭代周期都成立的模式，之后你才能确定地这么说。

### 当证据要求时就转向

万一即便投入了所有这些工作，你似乎就是无法达到产品—市场契合，怎么办？你的结果不支持你出发时的方向，这一事实并非失败，而是系统在起作用：MVP 阶段就是被设计来在你对错误答案过度投入之前，把这一信息浮现出来。

当数据不支持你当前的产品时，用 Claude 来梳理那些数据在告诉你什么。

- —探索其他客户细分。也许那些没有转化的用户从一开始就不是正确的目标。正确的受众往往已经在你的数据里，只是被低估了。
- —调整你产品的价值主张。也许你有正确的受众，但你的 MVP 就是没能引起用户共鸣。对入门流程、信息传达或核心功能侧重做一处调整，有可能在不改变你已构建之物的情况下修复它。

对"这种脱节也许深到需要一次更根本的改变"这一可能性保持开放。

练习：如果你已完成三个或更多迭代周期，却仍未朝产品—市场契合基准取得有意义的进展，那么在决定下一步做什么之前，用 Claude 跑一次诊断。把你的留存数据、你的用户反馈和你最初的问题假设喂给它，并问它三个问题：

- ·这份数据里是否有某个细分群体的反应与其余群体不同？
- ·设计价值与体验价值之间的差距，是定位问题还是产品问题？
- ·要让当前产品找到真正的 PMF，什么必须为真，而鉴于你所看到的，那个情景现实吗？

让答案来决定你是调整、转向，还是回到创意阶段。



# 第 5 章 上线阶段

如果说 MVP 阶段关乎证明你的产品值得存在，那么上线阶段关乎证明你的业务值得增长。

## 上线阶段的目标

在上线阶段，创业创始人必须把早期牵引力转化为一台可重复、可持续的增长引擎。除了让产品达到生产就绪，你还必须加固底层基础设施，同时围绕你的产品构建一家真正的公司。

在创意和 MVP 阶段，创业公司天然以创始人为中心，因为你需要完整的情境感知和紧密的反馈回路。但如今，仍试图亲手抓住每一根线头的创始人，会成为上线阶段的瓶颈。目标不是把你自己从公司里抽离，而是构建运营系统，腾出你的注意力，去做只有创始人才能做的决策。

## 上线阶段的退出标准

上线阶段的退出条件有三个要素：

1. 1增长是可重复且由渠道驱动的。你不只是在留住用户，你是在通过特定渠道以可预测的方式获取他们，且单位经济清晰：CAC、LTV 和回本周期都是你知道并能站得住脚地解释的数字。
2. 2产品能承受生产负载。基础设施已加固，安全与合规井然有序，可靠性在真实生产条件下成立（而不只是你测试过的条件）。
3. 3运营在没有创始人瓶颈的情况下运转。流程已存在，自动化已就位。你不再是那个亲自处理支持、分诊、冲刺规划或报告的人。

## 上线阶段的挑战

找到产品—市场契合是早期创业生命周期里最难的问题。如今，创始人的挑战变成了保持它。上线阶段是这样一个地方：找到了真实产品牵引力的公司，如果环绕并支撑产品的组织跟不上，仍可能分崩离析。以下是要警惕的失败模式。

### 技术债到期了

挑战：为速度和验证而构建的 MVP 代码库，跑得足够好以证明产品行得通，但生产流量、新功能和不断增长的复杂度，正在让那些捷径暴露出来。

在 MVP 阶段，积累一些技术债是为速度做的合理取舍。在上线阶段，那笔债开始计息，而它越久不被处理，修复就越昂贵。

解决方案包括：一次系统性的架构审计以识别结构性弱点，有针对性的重构以处理其中最糟的部分，以及一次有意义的测试覆盖扩展，使下一轮功能工作不会重新引入同样的问题。

### 创始人成为瓶颈

挑战：在 MVP 阶段，创始人身处每一个回路是一种资产。到了上线阶段，随着支持量增长、产品决策堆积、运营复杂度倍增，同样的本能变成了约束。

从"亲自做事"到"设计那些做事的系统"的转变，是创业生命周期里最艰难的转变之一。因为很少有一个清晰的时刻标志它发生，风险在于完全错过它，停留在构建者模式，而组织在你周围停滞。这正在发生的蛛丝马迹包括：本该一小时搞定的决策现在要等你一周才腾出手；支持请求堆积成山，因为只有你知道答案；以及运营任务只有在你个人记得去做时才会发生。

补救办法是对你亲自处理的一切——从最微小的任务到最高风险的决策——做一次彻底的审计，以识别什么可以被系统化、什么可以被委派，以及什么确实仍值得创始人的时间与注意力。

### 安全与合规不再可推迟

挑战：在 MVP 阶段把安全与合规措施保持简单是可以的，但现在，有了真实用户、真实数据，以及桌面上可能出现的企业合同，它就变成了一项负债。

在 MVP 阶段，只有寥寥数位测试用户、生产中没有敏感数据时，安全漏洞是理论风险。然而，一旦你的产品进入生产、有真实用户依赖它，那个假设就变成非常真实的暴露风险。此外，那些不适用于原型的合规要求，在你处理客户数据、处理支付，或向受监管行业销售的那一刻，绝对开始适用。

补救办法是在生产规模到来之前、而非之后，做一次系统性的安全与合规评审，并把浮现出的一切都当作必需的修复——而不是建议——在下一波用户到来之前完成。

### 在你准备好之前就扩张

挑战：新市场和融资机会看起来像增长机会。它们也可能是产品—市场契合走向死亡的地方。

你已建立的初始牵引力是真实的，但它也特定于你的早期受众。过早扩张进入一个与你原始市场有实质差异的市场，会引入新的用户行为、合规要求、支付基础设施，以及你的产品并非围绕其设计的基线预期。突然之间出现了太多新变量，你失去了清晰解读自己数据的能力。你还冒着在追逐一个全新且未经验证的受众时，忽视你原始用户群的风险。

## Claude 如何帮助上线阶段的创始人

Claude 的三种形态在上线阶段都被充分使用，且彼此支撑：每个工具产出的输出会成为另外两个的输入。结果会自然地复利累积，同时使用三个工具的创始人得到的远不止三者之和。

这正是超精简创业模型在结构上之所以可能的原因。当 Claude Code 构建产品、Claude Cowork 围绕它构建公司、而 Claude 帮助把产品与组织知识运营化时，一支小团队就能像一家其 n 倍规模的公司那样运转。

### 在技术债复利之前修复它

你的 MVP 代码库能工作，但它也需要一次系统性的修复，去搜寻任何可能变成结构性负债的技术债。

首先，用 Claude Code 跑一次完整的架构审计：识别代码库哪里脆弱、哪些捷径会越来越昂贵难以维护，以及哪里测试覆盖薄到下一轮功能工作会重新引入同样的问题。

把 Claude Code 的审计发现反馈给 Claude，去为修复工作分诊和排序：什么需要在下次发布前修好、什么可以等一个冲刺，以及在你当前阶段什么算作可接受的持续债务。这也是把你在 MVP 阶段做出的架构决策（那些因为没时间写下来而只活在你脑子里的决策）记录下来的时刻。现在把它们写进 CLAUDE.md，能确保未来每一次 Claude Code 会话都从对"系统如何设计、为何如此设计"的共同理解出发。

练习：指挥 Claude Code 审计你的 MVP 代码库，产出一份排好优先级的清单：结构性弱点、测试覆盖缺口和重构候选项。然后把这份清单喂给 Claude，请它跨你接下来的若干冲刺为修复工作排序：任何你需要先处理的重大问题、可以与功能开发并行处理的事项，以及可以等待的事项。

### 构建那些替代创始人注意力的系统

构建那些腾出你注意力、让你去处理只有创始人能搞定的职责的运营系统，要求精确知道你的注意力正流向何处。用 Claude Cowork 对你当前的运营负担做一次结构化审计，记录每一项重复任务、每一个落到你桌上的决策，以及每一个只因你个人记得去做才会发生的工作流。然后让 Claude Cowork 把这份清单分类为：可以完全自动化的、需要人但不一定是你的，以及确实需要创始人判断的。

审计完成后，用 Claude Cowork 为自动化候选项设计工作流逻辑：什么触发每个工作流、决策规则是什么、输出长什么样，以及完成后它去哪里。

### 把安全与合规做成一条产品工作流

用 Claude Code 浮现那些在 SOC 2、GDPR 或 HIPAA 审计中、以及你的目标市场要求的标准中经常出现的代码级问题。这会同时浮现漏洞和合规缺口。把这些发现喂给 Claude，帮你为修复工作排优先级，并设计企业买家在签约前会要求的控制措施、审计日志和访问管理。

注意：AI 扫描是一种辅助，但不能替代有资质的合规评审。

接下来，把合规工作流建进你的开发周期，而不是当作一次性项目来跑；合规文档需要持续维护和更新。对于正在接洽企业合同或国际市场的创始人，这也是 Claude Code 安全扫描能帮你为一次独立安全评估做准备的时刻。

练习：用 Claude Code 跑一次面向你目标市场所要求框架的代码级安全评审。把输出喂给 Claude，请它产出两样东西：一份排好优先级的安全修复序列，以及一份你需要产出、以满足潜在企业买家合规评审的文档与控制清单。

### 把你一直在跳过的产品管理流程立起来

上线阶段需要一套轻量、可重复的流程，无需创始人介入即可触发或运转。用 Claude 来设计你的产品时间线和工作周期将如何组织、在 Claude Code 动一个功能之前一份规格需要包含什么、缺陷报告如何分诊和路由，以及你的每周指标报告涵盖什么、如何分发。

流程设计完成后，用 Claude Cowork 来构建并运行运营层：安排冲刺仪式、把进来的缺陷报告路由到正确的地方、从你连接的数据源编制每周指标，并维护那条让用户信号持续流入产品决策的反馈回路。

练习：请 Claude 设计一套轻量的产品管理操作系统：一个确定的冲刺节奏、一个最小规格模板、一棵缺陷分诊决策树，以及一份从你真实数据源拉取的每周指标简报。然后设置 Claude Cowork 来实施并运行该系统中重复的运营要素，比如排期、路由和报告编制，让它们在没有你的情况下按时发生。




# 第 6 章 规模化阶段

在规模化阶段，创始人的角色从构建者重新聚焦为面向公众的高管。产品仍居核心，但你个人的日常工作越来越关于公司本身。即便你努力维持那种精简、以 AI 为中心的结构性优势，你的注意力也必须扩展到分析师简报会和 IPO 路演这类新的规模化阶段活动。

## 规模化阶段的目标

规模化技术基础设施的工作仍在继续，如今又加上了规模化组织本身、并把它催熟成一门生意的工作。

在规模化阶段，你面对的是从数千用户走向数百万，从一个市场走向许多市场。在此前的每个阶段，增长是你可以靠贴近用户、并根据紧密反馈回路的数据加上一份健康的创始人直觉来摸索前行的东西。但如今，目标是构建由成熟组织运营所支撑的系统性增长。

对一家 AI 原生创业公司而言，你的目标应是通过累积的深度构建一条可防御的护城河，它源自你构建进产品的专长、你的产品与用户依赖的其他工具和平台的集成深度，以及专有的系统数据和工作流。那些在一个方向上、在一致的基础设施上持续构建的创始人，如今手握真正难以复制的东西。

在这个阶段，公开市场投资者、分析师、监管机构、企业采购团队和收购方会施加更大的压力——以及更大的怀疑——因为现在赌注更高了。你的产品和组织必须经受外部审视：不只是你所构建之物的能力，还有环绕它的治理、合规姿态、财务控制和战略叙事。

## 规模化阶段的退出标准

规模化阶段的退出条件不再是单一里程碑，而是一个阈值事件：即便创始人越来越不直接运营日常事务，公司仍可持续。你已展示了系统性增长；构建了能让最苛刻的外部评审者满意的组织治理与合规基础设施；并且对"如果一个资金雄厚的在位者今天复制你的产品，你的用户会留下吗？"这个问题有一个扎实的答案。

实践中，这个阈值通常会取三种形式之一：在一个不再需要外部资本的规模上的可持续盈利、IPO 就绪，或被收购。三者都要求你的增长是系统性且可审计的、你的产品护城河经得起审视，以及你的组织在运营上成熟且可持续。

当这一切为真时，恭喜你：你的创业公司已经从一个赌注变成了一门生意。

## 规模化阶段的挑战

### 委派运营层

挑战：规模化阶段的运营系统必须可靠、可持续地运转，而无需有人时刻照看。对一位从第一天起就亲力亲为的创始人来说，那个转变既是结构性挑战，也同样是心理挑战。

你在上线阶段的工作是创建这些系统；到了规模化阶段，它变成了 (1) 把这些系统催熟到完全值得信任，以及 (2) 然后真的去信任它们。

这比听起来要难。即便你是一位擅长委派的创始人，什么该交出去、什么该留在自己手里也并不总是显而易见。交得太多、太快——尤其是交给 AI 自动化的系统——关键决策就会在缺少只有创始人能提供的关键上下文的情况下被做出。但抓得太久，你又会成为瓶颈。

这里根本的挑战，是识别那些只活在创始人脑中和未被记录的工作流里的机构知识，然后把它编纂成有文档、可审计、可转移的系统。

### 规模化技术运营

挑战：客户不再只评估你的产品；他们想知道你的组织能否成为一个可靠的基础设施伙伴。

创业前三个阶段的技术挑战集中在代码库上：在不积累技术债的情况下构建正确的解决方案，然后为真实用户加固安全与合规。到了规模化阶段，挑战如今变成了围绕代码库构建的一切；创建那些标志成熟度的支持基础设施、文档和可靠性保证。

签多年合同的更大规模客户和机构买家在签约前就想要这些，而且一旦签了，他们也会据此要求你。不过，把你带到这一步的那套同样的 AI 基础设施，能帮你构建有明确响应时间的专门支持职能，以及新客户工程团队真的能用的文档。

### 规模化组织职能

挑战：一家规模化阶段的公司通常需要招聘、薪资、会计和法务运营这类组织基础设施，无论由多少人在运营它。

在上线阶段，把运营系统化意味着把吞噬创始人注意力的工作流自动化。一家规模化阶段的创业公司如今需要发展出一组更广、在某些方面更具决定性的运营职能，比如财务报告、合规监控、合同管理和客户支持，仅举几例。

### 构建一个 GTM 职能

挑战：有机增长有天花板，而大多数规模化阶段的创始人在不得不构建一个真正的进入市场职能之前就撞上了它。

创意、MVP 和上线阶段的增长，往往源自创始人主导的销售，从一篇时机恰当的 Product Hunt 帖子到与早期客户的个人关系。然而这样的有机增长只在一定程度内奏效，大多数创业公司在规模化阶段撞上这个上限。迹象包括用户曲线变平、获客成本上升，以及一条只有创始人亲自参与才会推进的管道。

规模化阶段的增长要求构建一台专门的增长引擎，去触达你产品的新的、更广的受众。然而，大多数创业创始人之前可能从未运营过营销、销售和分析师关系项目这类事。一个正经的 GTM 动作不只要求建立新系统和流程，还要求创造一种品牌嗓音和故事，讲清你想如何谈论你的产品。因为在创业生命周期的这个阶段，你将需要它去触达的不只是个别新用户，还有投资者和企业买家这样的整个目标受众。

所幸，GTM 职能不必很大也能有效，而构建了产品的那套同样的 AI 基础设施，能把它引导着推向市场。

## Claude 如何帮助规模化阶段的创始人

早期创业阶段把 Claude 用作产品本身的根基性基础设施：验证创意的调研伙伴、设计并构建原型的工程团队，以及让单创始人创业成为可能的 AI 运营层。到达规模化阶段的 AI 原生创业创始人，如今可以用 Claude、Claude Code 和 Claude Cowork 以他们当初构建时同样的方式继续规模化。

### 把日常任务移交给 Claude Cowork

带着一份清醒的认识开启规模化阶段：你现在最需要把时间和注意力投到哪里——对从未做过生意的初次创始人来说，这本身可能是个挑战。Claude 能帮上忙，列出在这个阶段只有你才该做的事，其中可能包括产品叙事决策、董事会关系、企业大单，以及创始人对创始人的对话。不在那份清单上的任何事，都是委派或 Claude Cowork 自动化的候选项。

练习：用 Claude 为你当前的运营层做一张瓶颈地图：每一个经由你路由的工作流、决策和审批。现在，请 Claude 推演当你一周不在时，每一个会发生什么。会卡住的那些工作流，正是你仍亲力亲为到足以拖累进展的地方。

这些与你和 Claude 一起做的创始人优先级与职责清单如何对应？

接下来，是时候压力测试你已经构建的系统，是否真的准备好随业务增长而规模化了。

练习：用 Claude 绘制你当前的工作流，然后问它当你一周不在时每一个会发生什么。会卡住的那些工作流，正是移交标准、升级路径或异常处理仍需收紧的地方。Claude 能帮你分析失败点并推荐适当的修复，让你按需更新或替换 Claude Cowork 的自动化。

### 把技术运营规模化为企业级基础设施

随着你规模化，买家需要确信你的产品和你的组织能作为长期基础设施被信任。代码库内部的技术工作一如既往地继续，但现在也有围绕代码库的技术工作要处理。

第一步是把机构知识转化为一个可规模化的系统。用 Claude 起草并维护企业采购期望看到的成文基础设施，包括产品文档、支持操作手册和 SLA。

与此并行，指挥 Claude Code 针对企业合同所要求的具体可靠性和安全标准审计并加固代码库，并构建出基于 Discord 的社区支持从来不必提供的技术支持基础设施：日志、监控、事件响应工具，以及让 SLA 真正可强制执行的可观测性层。

然后 Claude Cowork 运行企业支持本身的运营层：工单路由、升级工作流、由产品变更触发的文档更新、续约追踪，以及企业客户成功所依赖的报告节奏。这三者合起来，给一支小团队带来一个大得多的组织的支持姿态，而这正是签一份多年企业合同要求你展示的。

练习：挑出你最苛刻的三个潜在客户，或确定三个你很想签下的理想客户。请 Claude 产出一份差距分析：在签多年合同之前，这些客户各自的企业采购团队期望看到什么样的文档、SLA 和支持基础设施，而你目前在哪里有差距？用这份输出去跨 Claude Code 和 Claude Cowork 为技术和文档工作排序。

### 构建一个真正的 GTM 职能

创始人的拼劲把你带到了这一步，但规模化你的创业公司要求创建并实施一套真正的进入市场策略。AI 能帮你构建，然后运行，那台完整的 GTM 引擎。

Claude 能从零协助构建根基性的 GTM 资源：市场细分、信息传达架构、分析师关系策略、销售操作手册，以及一旦你开始与公开市场投资者、企业买家和华尔街分析师对话就至关重要的、面向投资者的指标叙事。这些受众各有自己的词汇，并以各自的标准来评估你；Claude 的工作是把你产品的价值主张翻译成一种对每个受众细分都切题的产品营销方法。

如今，Claude Cowork 能成为你的战术执行层：内容管道、外呼序列、分析师简报会的事务、新闻编辑室和公关节奏、CRM 卫生、管道报告，以及把 GTM 策略变成实际商业动作的众多重复周期。

在 GTM 动作要求产品营销基础设施的地方——交互式演示环境、集成文档、沙箱租户、API 参考、技术单页——Claude Code 能为你构建。买家期望在技术上评估你的产品，而在规模化阶段，一段 Loom 视频和一份销售 PPT 已不再够用。这也是让你的 GTM 动作可异步运行的基础设施：一个构建良好的演示环境会在你开董事会时帮你拿下交易。

### 把领域专长和机构知识变成 AI 上下文

许多超精简创业创始人，正在为一个他们在某个特定行业亲身经历或第一手观察到的真实问题，构建高度具体的应用或工具。智能体 AI 如今让从未写过一行代码的创始人，也能用他们的领域专长去构建解决复杂问题的产品。Claude、Claude Code 和 Claude Cowork 各自在把创始人知识转化为复利累积的产品独特性中扮演一部分角色。

用 Claude 来捕捉、组织并打磨创始人知识，把领域专长放到产品能够触及的地方。通过延展的对话、项目和记忆，创始人可以把他们知道的一切——行业黑话、监管陷阱、边界情况、痛点、为什么这个问题显而易见的答案行不通的原因——分享进一个结构化、可检索的上下文。技能（Skills）随后能把重复的工作流（例如"我如何审计一份商业租约"、"我如何分诊一份患者接诊表"）编纂成可复用的例程，Claude 每次都以同样的方式运行它们。数月之后，这会成为一个没有任何通用 AI 能匹敌的专有知识底座。

用 Claude 把你的领域知识外化，对于把行业特定的边界情况编码进你的产品而言变得极有价值：举例来说，一个通用 AI 医疗账单工具会在 340B 药品计划报销上出错，而你的工具对它们有专门逻辑。Claude Code 帮你把你所在领域其他专业人士经历的常见痛点，翻译成校验逻辑、提示词改进，或与一个你竞争对手闻所未闻的小众行业系统的 MCP 集成。结果是，你的应用或工具的深度和广度都以一种竞争对手根本无法复制的方式持续复利累积。

练习：找出一个通用竞争对手在你所在垂直领域里肯定会搞错的边界情况。与 Claude Code 协作，基于一个你实际见过的场景为它构建一个专门的测试用例（不是单元测试）。每当一个类似的边界情况浮现，就把它加进去。你的测试套件会成为你护城河的一张地图。

### 把累积的用户数据复利成可防御的优势

随着用户与你的产品互动，他们产生行为信号（即他们接受哪些输出、拒绝哪些），这为产品路线图提供信息。随时间推移，你会了解你这个特定用户群的具体模式、偏好和边界情况。这就是我们所说的复利价值：每一次改进让产品更有用，从而带来更多使用，从而创造更多反馈，从而带来更多改进。

这些数据是被时间锁定的、上下文特定的，模仿者无法重建：你根本买不到成千上万名在你产品内打磨自身工作流的用户的行为指纹。

Claude 能帮你审计你已收集的任何用户交互数据、识别其中信号最强的行为模式，并设计那条把持续使用变成系统性模型改进的反馈回路。

练习：把你产品交互数据的摘要喂给 Claude：你一直在收集什么、收集了多久，以及你对用户随时间如何与产品互动了解多少。请它找出该数据中信号最强的三个行为模式，并设计一条把每一个都变成系统性模型改进的反馈回路。然后请它帮你起草一份一页纸的护城河叙事来指导产品营销：你的数据飞轮如何运作、它已转了多久，以及为什么一个今天才起步、资源雄厚的竞争对手在两年内无法复制它的故事。

### 创造工作流锁定

复利的数据网络效应让你的产品更难复制，而用户工作流锁定让你的产品更难离开。用户在他们的日常运营中运行你产品越久，它就越深地嵌入他们实际的工作方式。他们已经在它之上构建了自动化、训练了人去使用它，并把它连接到他们的数据源和其他工具。他们开发的提示词、他们打磨的工作流、他们标准化的输出，全都是围绕你产品做什么、怎么做而塑造的。到了这一步，更换就从一个产品决策变成了一个全规模的运营项目。

创造工作流锁定的第一步，是请 Claude 按集成深度绘制你当前的客户群。对每个客户细分，识别他们在你产品之上构建了什么工作流、依赖哪些集成。这显示出你的产品在哪里黏住了，以及它需要在哪里扎得更深。

你提供的集成越多，客户能用来构造依赖你产品的工作流的界面面积就越大。Claude Code 帮你快速搭起与你目标用户依赖的数据管道、项目管理工具和其他系统的原生集成。Claude Code 还能构建那些让客户不只是使用你的产品、而是在它之上构建的 API、webhook 和 SDK——锁定的最深形式。

练习：请 Claude 帮你为你前十大客户构建一份工作流集成审计。对每一个，记录他们已构建的自动化、他们依赖的集成、贯穿你产品运行的团队工作流，以及你对他们更换成本的估计。然后请 Claude 找出这一群体中的模式：哪些类型的集成为你这个特定产品创造了最深的锁定，以及对那些目前还停留在表面的客户，你可以构建或开放什么来加深集成。




# 第 7 章 工作没变，规则已变

在 AI 领域，创始人的工作没有改变：找到一个真实的问题，构建能解决它的东西，并把它规模化成一家重要的公司。改变的是抵达那里的路径。横跨四个阶段——创意、MVP、上线和规模化——AI 把数个季度压缩成数周。

过去要数月的验证周期，现在只要一个下午。一个能跑的原型不再需要一位拥有合适技术栈的联合创始人；它只需要一个清晰的问题，以及与一个编程智能体的几次聚焦会话。上线就绪从上线前的手忙脚乱压缩成一条持续的工作流。而在规模化时，过去逼着早期员工去当救火队员的运营重担，越来越可以被移交给 AI，腾出你的团队把注意力花在那些会成为你护城河的判断性决策上。

瓶颈不再是你能构建什么，而是你选择构建什么。
