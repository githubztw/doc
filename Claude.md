# 数据保存
| **路径**                  | **内容**     | **说明**                                               |
| ------------------------- | ------------ | ------------------------------------------------------ |
| `~/.claude/settings.json` | 全局配置文件 | 更新通道、API Key、环境变量、MCP Server 等所有全局设置 |
| `~/.claude/CLAUDE.md`     | 全局记忆文件 | 跨项目生效的指令和偏好（后续"记忆管理"章节详细介绍）   |
| `~/.claude/projects/`     | 项目级数据   | 每个项目的会话记录、自动记忆、本地配置等               |
| `~/.claude/credits.json`  | 用量与额度   | API 调用次数、Token 消耗统计                           |

除了全局配置，每个项目根目录下也可以创建 `.claude/settings.json`（项目级配置）和 `.claude/settings.local.json`（项目私有配置，不提交到 Git）。三者的优先级为：**项目私有 > 项目共享 > 全局**，后续在模型替换章节会详细说明。

# 更新策略

| **通道** | **配置值** | **更新节奏**         | **适用场景**                             |
| -------- | ---------- | -------------------- | ---------------------------------------- |
| 最新通道 | `"latest"` | 新版发布后立即推送   | 想第一时间体验新功能、能接受偶尔的小 Bug |
| 稳定通道 | `"stable"` | 比 latest 延迟约一周 | 正式项目开发，优先保证稳定性             |

```josn
{
  "autoUpdatesChannel": "stable"
}
```

# 常见命令



直接更新: `claude update`

环境检查：`claude doctor`

```
任何时候 Claude Code 的行为不符合预期，先跑一次 claude doctor。它相当于 Claude Code 的"体检报告"，80% 的常见问题都能在这里找到线索。在向社区求助或提交 GitHub Issue 时，附上 doctor 的输出也能显著加速问题定位。
```



双击`CTRL+C`即可退出对话，同时可以看到当前对话的ID

```
claude --resume 7d46f50c-c227-4630-ab21-838dc9226143
```



恢复上一次停止的会话：`claude -c`

> 使用claude -c必须在同一文件夹内，恢复对话才会生效



清空当前对话 `clear`

> 上下文窗口被释放，相当于重置到了接近初始状态。
>
> 以避免在长会话中接近上下文限制时出现的问题。如果你觉得当前会话已经变得卡顿或响应质量下降，可以用 `/clear` 来清理。

压缩对话历史 `/compact [指示]`

恢复到指定会话 `/resume [ID|名称]`

切换模型 ：  `/model [模型名]`

进入Plan mode：`/plan`

初始化项目，生成 `CLAUDE.md`：`/init`

当前上下文使用情况：`/context`

查看未提交的变更：`/diff`

回滚对话和/或代码到历史检查点：`/rewind`

复制上一条 AI 回复到剪贴板 `/copy`

运行途中快速补充信息或提问 `/BTW`

设置模型思考深度：`effort [high|low]`

导出对话为文本文件：`/export`

审查OR,提交代码前质检：`/review`

> 在cursor中定义规则： 放在.cursorrules目录下

## Claude Code 能否"多开"

**不同项目目录——完全可以多开**

实例的会话、记忆、Checkpoint 各自独立，互不干扰。这正是"项目级隔离"设计的优势——每个项目目录映射到 `~/.claude/projects/` 下不同的文件夹，数据天然隔离。

 **同一项目目录——强烈不建议多开**

虽然技术上 Claude Code 不会阻止你在同一个目录下启动第二个实例，但这样做会引发一系列问题：

是的，`/clear` 命令会清空当前会话的对话历史，从而释放上下文窗口（context window）中的 token 占用。

具体来说：
- 清除所有之前的对话轮次
- 只保留系统提示（system prompt）和当前这一轮的消息
- 上下文窗口被释放，相当于重置到了接近初始状态
