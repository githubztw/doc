##  1. 成本
模型选择高低搭配

### 1.1 上下文管理是成本控制的核心

智能体与上下文隔离：
1. 提升信噪比的技术收益。

2. 子智能体绝非锦上添花的奢侈品，而是至关重要的经济决策工具。

   > `Claude API`是无状态的，每次调用都必须携带完整的对话历史。若在主对话中直接读入一个5000行的日志文件，这5000行内容将在后续每次API调用中被重复发送，都在为那5000行日志重新付费

3. `CLAUDE.md`的精简

   > 该文件在每次API调用时也都被自动加载，对于详细文档，应移至单独文件中按需读取，或转化为Skills仅在特定场景下触发。

`/cost `:查看当前会话的详细成本统计

### 1.2 `Prompt缓存`

为最大化缓存命中率，需要遵循以下3个关键策略

**保持`CLAUDE.md`稳定**：频繁修改配置文件会导致前缀变更，从而致使缓存失效。

**会话内聚合任务**：尽量在同一会话中完成相关联的任务，避免为每个小任务都开启新会话

**延续既有会话**：善用--resume或--continue参数延续现有会话，切勿每次操作都从头启动新会话。

| 参数         | 简写 | 核心作用                         | 执行逻辑                                                     |
| ------------ | ---- | -------------------------------- | ------------------------------------------------------------ |
| `--continue` | `-c` | 直接**恢复当前目录最近一条会话** | 读取项目内 `last-session` 标记，无交互弹窗，一键接续上次对话 |
| `--resume`   | `-r` | 灵活恢复指定会话                 | 1. 不带 ID：弹出交互式会话选择列表；2. 携带 `会话ID/会话别名`：精准打开目标会话 |

## 2. 调试

### 2.1 --debug模式

简写`-d`，开启全量底层调试日志输出，打印 Claude Code 内部完整运行链路细节，专门用来排查：

+ Hook 不执行
+ MCP 连接失败
+ Skill 不触发
+ 工具被拦截
+ API 报错
+ 上下文溢出
+ 权限阻断等静默异常

支持过滤模块：`api`、`mcp`、`hooks`、`tool`、`skill`、`agent`、`file`、`statsig`。

```bash
# 仅打印 MCP + Hook 日志（排查MCP和钩子首选）
claude --debug mcp,hooks -p "分析代码漏洞"

# 只看API请求，屏蔽文件读写日志
claude --debug api,statsig,!file
```

`statsig`：Claude 内置统计 / 埋点模块，输出会话指标、工具调用次数统计、上下文窗口占用、功能触发打点、性能耗时指标。

> 适用场景：排查会话卡顿、token 溢出、功能未触发统计。

**日志持久化**：`--debug-file`

```bash
# 把调试日志写入文件，不刷屏终端：
claude --debug --debug-file ./claude-debug.log -p "执行全量代码审计"
```

在交互会话中遇到异常时。只需要按下Ctrl+O快捷键，即可随时切换verbose（详细）模式。可话过程中实时查看Hook的执行进度、工具调用的详细参数与返回信息以及模型当前的思考路径。

### 2.2 stream-json：实时透视“思考流”

在Headless模式下，`--output-format stream-json`是将Claude的决策过程转化为机器可读数据。获取更细粒度的流式数据，包括正在生成中的文本片段

```bash
# jq '.'作用是格式化、美化打印 JSON
claude -p "分析src/目录的架构问题" --output-format stream-json | jq '.'
```

当遇到:

+ “Claude为什么在某个点突然停下来了？”
+ “它是否在死循环中重复生成相同内容？”

### 2.3  PostToolUse Hook：构建持久化审计日志

```bash	
#!/bin/bash
# .claude/hooks/audit-log.sh
INPUT=$(cat)
TOOL=$(echo "$INPUT" | jq -r '.tool_name')
TIMESTAMP=$(date -u +"%Y-%m-%dT%H:%M:%SZ")

# 输出拼接格式UTC时间 | 工具名 | 工具入参JSON到audit.log文件中去
# 如：2026-06-25T08:36:10Z | Write | {"file":"src/login.js","content":"..."}
echo "$TIMESTAMP | $TOOL | $(echo "$INPUT" | jq -c '.tool_input')" >> .claude/audit.log
exit 0
```



**配置`setting.json`**

```
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "*", //所有工具都触发
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/audit-log.sh"
          }
        ]
      }
    ]
  }
}
```

### 2.4 会话存档与断点调试

Claude Code将所有会话记录存储于`~/.claude/projects/{project}/{sessionId}.jsonl`。若涉及子智能体，其独立记录则保存在`subagents/agent-{agentId}.jsonl`中.

当自动化执行出现异常时，你可以利用`claude --resume <session-id>`精确“时光倒流”至故障发生的会话现场。这将使你能够检查Claude在该时刻所接收到的具体信息，以及复盘Claude基于当时信息所作的判断与工具调用逻辑。

### 2.5 异常行为诊断思路

#### 2.5.1 上下文缺失问题

**原因**：Claude无法获取完成任务所需的关键信息。

**典型症状**：反复读取同一文件、回答问题空洞或明显缺乏上下文依据。

**诊断方法：**使用--debug，检查Claude实际接收到的文件内容和上下文范围。

**解决方案：**在CLAUDE.md中补充清晰的项目导航指引，或在Prompt中直接提供必要的上下文信息与代码片段。



#### 2.5.2 指令冲突

**根本原因：**CLAUDE.md中存在两条或多条相互矛盾的规则。

**典型症状：**行为在两种模式间摇摆不定，或直接忽略部分既定规则。

**诊断方法：**全面审查CLAUDE.md中的所有规则，识别逻辑互斥点。

**解决方案：**明确规则优先级。使用显式声明消解歧义，例如，“当规则X与规则Y冲突时，优先遵循规则X”。



#### 2.5.3 工具权限不足

**根本原因：**Claude试图执行某项操作，但缺乏对应的工具支持或权限授权。

**典型症状：**反复尝试某操作后失败，或被迫使用不合适的替代工具完成任务。

**诊断方法：**检查--allowedTools配置项，并查阅Hook的拦截日志，以确认是否有请求被阻断。

**解决方案：**开放必要的工具权限，或调整任务描述，引导Claude使用当前可用的替代方案。



调试Claude与调试代码的本质区别：

+ 代码执行是确定性的，相同输入必然产生相同输出；而Claude的行为具有概率性，相同的Prompt在不同上下文中可能引发截然不同的反应。

  因此，无法依赖传统的“复现bug”模式来解决问题，必须转而依靠日志分析、审计轨迹回溯以及`CLAUDE.md`规则审查，以此推断问题产生的根本原因。





## 3. 文件访问控制

1. `.claudeignore`文件可用于阻止Claude索引特定文件，在某些场景下有被绕过的风险
2. `permissions.deny`配置中显式禁止对敏感文件的读取操作,更安全可靠

**建议：**同时使用这两种机制，构建双重防护：.claudeignore作为第一层优化，减少不必要的文件索引开销；permissions.deny作为第二层硬性防线，确保即使索引被绕过，敏感文件也无法被读取。

### 3.1 权限管控

`--allowedTools`和`--disallowedTools`参数支持模式匹配

1. 在命令中使用

   ```bash
   # 场景1：CI/CD流水线中的代码审查（仅需要读取权限）
   claude -p "审查 PR 变更" --allowedTools "Read" "Grep" "Glob"
   ```

2. 在团队共享的`.claude/settings.json`文件中，可通过deny（拒绝）、allow（允许）和ask（询问）3级规则进行精确控制

   **设置permissions**

   ```json
   {
     "permissions": {
       "allow": [
         "Bash(npm run lint)",
         "Bash(npm run test *)",
         "Bash(git commit *)"
       ],
       "deny": [
         "Bash(curl *)",
         "Bash(wget *)",
         "Read(./.env)",
         "Read(./.env.*)",
         "Read(./secrets/**)",
         "Bash(git push --force *)"
       ],
       "ask": [
         "Bash(git push *)"
       ]
     }
   }
   ```
       权限规则优先级顺序：deny→ask→allow。这意味着如果某项操作同命中了low、deny规则，操作将被直接拒绝。

### 3.2 忽略文件`.claudeignore`

项目根目录新建 `.claudeignore`，语法完全兼容 `.gitignore`，规则为**排除匹配到的文件 / 目录**。

1. 执行 `Read`、目录扫描、自动项目索引时，直接跳过忽略列表内文件；
2. 不会把忽略文件送入 LLM 上下文，减少 token 占用、提升工具响应速度；

**注意：**仅作用于**文件自动索引**，如果手动用绝对路径强制读取被忽略文件，仍可读取；`.claudeignore` 是轻量化性能优化，不是权限拦截。

```bash
# 编译产物
dist/    # 递归忽略目录内所有内容
build/*  # 只是忽略一级直接子文件/子文件夹，不递归深层
out/
*.log

# 大型二进制、缓存
*.zip
*.tar.gz
.cache/
.venv/
```



## 4. 大型代码库：在有限上下文窗口中驾驭百万行代码

### 4.1 `CLAUDE.md`

`CLAUDE.md`应被视为核心代码，纳入严格的代码审查流程。

1. 要层次化`claude.md`，这种机制实现了规则的“按需激活。
2. 将专业领域的详细知识迁移至Skills模块，实现按需加载。这种架构设计能显著提升效率。

**CLAUDE.md编写原则**

+ 大模型无法自行推测关键命令：构建、测试及部署的具体指令。
+  定制规范：与默认设置不同的代码风格规则。
+  架构约束：项目特有的架构决策与设计限制。
+  环境要求：开发环境的特殊配置（如必需的环境变量、非标准工具链等）。
+  避坑指南：常见陷阱及不易察觉的系统行为。

**应避免**
+ 可推导信息：Claude通过阅读代码即可自行推断的内容。
+  通用惯例：标准的语言规范（Claude已内置相关知识）。
+   易变信息：频繁变动的内容（此类信息会导致缓存失效）。
+   文件详述：文件的代码描述（应交由搜索工具处理）。
+   空泛建议：如“编写整洁代码”等不言自明的原则。

### 4.2 引导搜索，而非全量阅读

Claude默认可能会尝试读取所有相关文件的全部内容，这不仅效率低下，还会显著增加成本。因此，必须在`CLAUDE.md`中制定清晰的导航策略

```
## 代码库导航原则
在处理涉及多文件的任务前，请严格遵循以下步骤。
1.使用 Grep 搜索关键函数名、类名或错误码，精准锁定相关文件范围。
2.利用 Glob 列出目标目录的文件结构，快速把握整体组织逻辑。
3.仅 Read 最相关的核心文件的具体内容。
4.严禁在无明确目标的情况下进行大范围文件读取。
5.对于可能产出大量输出的任务（如日志分析、大文件搜索），应委托给子智能体处理
```



### 4.3 精准表达

1. 量化一切，避免模糊描述，务必使用具体数据

   + 处理大量数据 —> 处理100万条记录
   + 响应要快”可写作—> P99响应时间< 200 ms
   + 代码要简洁 —> 函数不超过20行，参数不超过4个

2. 示例驱动，对于复杂需求，一个示例胜过10句描述

   ```
   需求：将驼峰命名转换为蛇形命名
   
   示例:
   输入: "getUserName" → 输出: "get_user_name"
   输入: "HTTPRequest" → 输出: "http_request"
   ```

3. 明确边界，定义**禁止项**与定义**执行项**同等重要，要防止Claude越界。

   ```
   范围限制：仅修改 auth 模块，严禁触碰其他文件。
   工具约束：必须使用项目现有的 logger，禁止使用 console.log。
   依赖管理：严禁添加任何新依赖。
   失败处理：若遇到无法解决的问题，请立即报告并停止操作，切勿反复尝试。
   ```

4. 分步确认

   针对复杂任务，分步执行远胜于一次性提交。通过将大需求拆解为独立的验证闭环，可显著降低偏差风险。

   ```
   第一步：“请分析这个 bug 的根本原因，暂不修改代码。”
   （待确认分析逻辑正确后，进入第二步）
   第二步：“基于上述分析修复 bug，仅修改必要代码。”
   （待确认修复方案无误后，进入第三步）
   第三步：“为此次修复编写对应的测试用例。”
   ```
