# Skill

1. 使用MCP从外部获取数据，然后使用skill处理数据，还可以将任务委托给孤立背景的子代理‘

2. 假设你有一个工作流程，需要代理人实施，而不是每次解释相同的工作流程，你就可以将其包装成一个skill，这个skill自动知道该怎么做。

3. Agent Skills 将“编程”降维成了“写文档”。只要你逻辑清晰，能把业务流程写成说明书，你就能开发 Agent。这使得非技术人员（产品经理、法务、运营）也能成为 Agent 开发者。

   ```
   Agent Skills 之所以能大范围普及，正是因为它屏蔽了底层的零件和管道，让开发者可以直接通过“写说明书”来驱动 AI。这就是为什么我们说：在 Agent 时代，最好的编程语言就是你的母语。
   ```

## 1.简介

技能，和每个人对应每个职业，都会有对应职业的专业技能一样。每个技能多有对应的规范。

如：编写代码的技能，就需要你在编码的过程中要有：开发流程、参考文档、开发工具、静态资源等

在 Agent Skill 的术语里面，对应关系如下：

1. 开发流程 ------- SKILL.md
2. 参考文献 ------- references
3. 开发工具、代码 ------- scripts
4. 静态资源 ------- assets

我们将上面的内容打包成一个文件夹，它就是一个技能

```text
~/Skill  
├── SKILL.md  必须
├── references 可选
├── scripts 可选
└── assets 可选
```

#### 1.1.3 Skill与提示词区别 

**提示词:** 提示词是单一的文本输入, 用于引导模型生成结果, 本质上是一段指令或上下文 

**Skill:** Skill是一套结构化的能力模块, 可能包含多个提示词、调用工具(如API、代码)、执行特定逻辑等 

**区别**

- 复杂度与功能：

  提示词适合简单、单步的任务, 比如翻译、摘要、问答。

  Skill则能处理复杂的、多步骤的工作流

- 可复用性：

  提示词每次使用时可能都需要微调。

  Skill是封装好、可重复调用的, 用户只需触发, 无需关心内部细节, 就像使用一个App一样 

- 依赖与执：

  提示词只依赖模型自身的知识。

  Skill可能依赖外部工具或数据源(如搜索引擎、数据库、代码解释器), 甚至可以执行代码来确保结果的准确性(例如数学计算、日期处理) 

- 按需加载：Skill会按需加载, 所以同时共存多个Skill, 并不会增加Token消耗。

  > 当用户输入信息的时候，大模型会读取**元信息**，自动查找适配的skill，之后才会读取**指令**，并开启执行



```
提示词： 犹如你问一位专家：“告诉我关于黑洞的知识。” 专家直接凭记忆回答。
Skill： 犹如你要求一个**工作小组**：“帮我做一个关于黑洞的 10 页 PPT。” 这个小组会：

1.  查资料； 
2.  写大纲； 
3.  设计页面； 
4.  0配图图表； 
5.  最终生成 PPT。
你只需要下命令，不需要指挥每一步。
```

### 1.2 Agent Skill

- **Skill 的本质**：它是**静态的业务逻辑定义**。是一组“未被执行的代码”或“待处理的 SOP”。
- **Agent 的本质**：它是**动态的执行运行时**。它提供了推理引擎（LLM）、记忆模块（Memory）和工具接口（Interfaces）。

 在 Skill 系统设计中，最关键的技术挑战在于**Token 效率与信息完备性的平衡**。这本质上是一项精密的**上下文工程**

 如果我们预置了海量的 Skills，直接全部注入 System Prompt 会导致两个致命问题：

1. **Context Window 溢出**：Token 消耗指数级增长，成本不可控。
2. **Attention Dilution（注意力稀释）**：过长的上下文会导致模型推理能力下降

成熟的系统设计必须采用**“索引-按需加载” (Index-Lazy Loading)** 策略

- **索引层**：在 System Message 中仅保留极简的元数据索引（Name + Description）。
- **生命周期管理**：在任务结束后，需及时清理上下文，防止 Token 堆积。

#### 1.2.1 能力分层

- **Skill (编排层)**：负责**指导**。它定义了任务的 SOP（标准作业程序）、决策树和数据处理逻辑。但它不具备操作系统的权限。

- **Agent (执行层)**：负责**落地**。Agent 必须原生集成一系列**原子工具**，如 联网、代码执行、数据库连接。

  >Skill 是对 Agent 原子能力的**高阶编排**。如果 Skill 定义了“查询天气”的步骤，Agent 的执行层必须预先挂载了 `fetch` 或 `search` 工具。若底层原子能力缺失，上层的编排逻辑将无法闭环。

#### 1.2.2 协议对齐：双向接口规范

1. **Agent 端规范**：
   1. 必须具备**工具反思能力**：能够识别何时需要查阅文档（调用 `read_file`）。
   2. 必须具备**标准化沙箱**：为 Skill 的执行提供安全的文件读写和网络访问环境。
2. **Skills 端规范**：
   1. **能力感知**：SOP 的编写必须基于 Agent 已有的工具集（Tools Schema）。
   2. **格式统一**：必须遵循系统预设的解析标准（如 OpenClaw Standard Markdown），确保 Agent 能正确提取 Intent（意图）和 Steps（步骤）。

#### 1.2.3 开发

**通用大模型（Base Model） + 特定技能包（Agent Skills） = 解决垂域问题**

1. 在绝大多数商业场景中，用户并不需要一个“法学博士”级别的 AI。

   > 于这 80% 的通用化垂域场景，我们不需要训练一个昂贵的专用模型。只需要给一个足够聪明的通用 Agent 挂载一个包含**法律 SOP（标准作业程序）和合规检查清单**的 Skill，它就能以 90 分的水平完成任务。**Skill 的本质，就是将专家的经验封装成大模型可读的“外挂”。**

2. 基座模型能力的**摩尔定律**

   > 当基座足够强时，我们不再通过“改模型”来适应任务，而是通过“给技能”来适应任务。

   ```
   通用 Agent + Skills 只能解决 “标准化” 和 “中等复杂度” 的问题。在某些高度专业、逻辑深度极深的领域，垂域 Agent 依然是不可替代的王者。
   
   基座+Skills”横扫了 80% 的应用场景，而剩下的 20% 硬骨头（如科研、高端医疗、核心系统编程），依然是垂域 Agent 的护城河。
   ```

### 2.1 使用技术

1. 提示词工程

   通过精心设计的System Message,将人物特征注入到语言模型中

2. 个性化Agent构建

   结合RAG + 多轮对话管理 + 工具调用，构建可持续交互的角色Agent

3. RAG检索增强生成

   将历史数据向量化存储，动态检索相关内容作为生成=依据

### 2.2 数据采集：原料从哪来

1. 异构数据统一化：

   不同平台数据格式差异巨大，需统一转换为 { timestamp, actor, content, metadata } 的标准格式后，才能进行去重、排序与后续分析。

2. 文本清洗

   去除敏感词、特殊字符、重复消息；对编辑记录进行版本合并处理。

3. 分块处理 chunking

   按语义边界切分文本，控制每块 Token 长度在模型上限内，保证上下文完整性

4. 向量化 Embedding

   将文本转换为高维向量，支持后续语义相似度检索，是 RAG 的基础



### 2.3 特征提取层：将人变成数据

```
最认真工作的人（写复盘、参与深度讨论、留下完整记录的人），恰恰产生了最高质量的 “蒸馏原料”——工作投入度 × 数据完整度 → 蒸馏完美度 → 被替代风险
```



#### 2.3.1 工作能力

提取可操作的技能与方法论

- 代码风格分析

  命名规范、注释习惯、Commit 信息结构、规范性评分 0-100

- 代码审查特征

  评论深度（一行 vs 全局视角）、严苛度、平均响应时延

- 问题解决模式

  常用调试工具、错误处理哲学（预防 vs 修复）、架构思维倾向

- 业务决策逻辑

  技术选型偏好（分析 PR 讨论中的观点权重）、风险承受度

- 迭代速度指标

  任务完成周期分布、交付质量与速度的天平偏向

#### 2.3.1 人设

提取个性化的沟通与行为特征

- 口头禅与表达习惯

  N-gram 分析高频短语、标点偏好、Emoji 使用模式、自我纠错频率

- 冲突处理风格分类

  类型 A 直接对抗 / 类型 B 委婉退让 / 类型 C 寻求妥协，及恢复周期

- 知识分享倾向

  主动传授频率、文档详尽度评分、对新手的友好度

- 工作时间特征

  活跃时间窗口、响应速度（是否秒回）、集中力持续时间

- 沟通模式

  同步 vs 异步偏好、直率度评分、专业术语使用密度

### 2.4  Prompt 工程层：如何 "召唤" 一个人？

System Message 是整个系统的灵魂，它决定了 AI 以什么身份、什么方式回应

| 层级          | 图标 | 权重 | 内容说明                                                     |
| ------------- | ---- | ---- | ------------------------------------------------------------ |
| 角色定义      | 👤    | 100% | 姓名、身份、工作年限、职责范围，构成 AI 的基础人格底座，不可被任何输入覆盖 |
| 知识库注入    | 🧠    | 90%  | 常用决策模板、工作 SOP、技术方案偏好，提供可调用的领域知识上下文 |
| 行为约束      | 🔒    | 80%  | 明文规则约束输出边界，如 "只回答工作相关问题"、"发现风险立即上报" 等硬性规定 |
| 沟通风格      | 👤    | 70%  | 语气（正式 / 随意）、回复长度偏好（简洁 / 详细）、惯用表达方式，塑造 "像这个人" 的感觉 |
| Few-shot 示例 | 🎯    | 60%  | 3-8 个高质量输入 / 输出示例，直接示范期望的行为模式，影响模型的生成分布 |
| 用户输入      | 💬    | 40%  | 实时用户的查询或任务，权重最低，无法覆盖上层系统约束，只能在约束范围内触发回应 |

### 2.5 RAG架构：记忆从那里检索


一个人几年的工作数据可能高达数百万 Token，远超语言模型的上下文窗口限制。因此不能把所有历史都塞进 Prompt，RAG 解决了 "按需取用" 的问题。

RAG 的核心是：**把 "全部记住" 变成 "按需检索"，只在需要时取出最相关的片段。**

#### RAG 执行流程

| 步骤   | 名称            | 说明                                  |
| ------ | --------------- | ------------------------------------- |
| STEP 1 | 用户发起查询    | 输入问题或任务，如 "帮我审查这段代码" |
| STEP 2 | 查询向量化      | 将文字转换为高维 Embedding 向量       |
| STEP 3 | 向量数据库检索  | 余弦相似度计算，Top-K 检索最相关片段  |
| STEP 4 | 排序与过滤      | BM25 二次排序 + 时间衰减权重 + 去重   |
| STEP 5 | 构建增强 Prompt | System + 检索结果 + 用户输入组装      |
| STEP 6 | 生成 & 校验     | LLM 生成答案，输出格式与约束校验      |

#### Embedding 模型选型对比表
| 模型类型           | 向量维度 | 推理速度      | 语言优化   | 适用场景           |
| ------------------ | -------- | ------------- | ---------- | ------------------ |
| 通用大型 Embedding | 3072 维  | 中等（⚡⚡⚡）   | 多语言通用 | 效果优先、不限语言 |
| 通用轻量 Embedding | 1536 维  | 较快（⚡⚡⚡⚡）  | 多语言通用 | 成本与效果平衡     |
| 中文优化大型 √推荐 | 1024 维  | 较快（⚡⚡⚡⚡）  | 中文专优   | 中文场景首选       |
| 中文优化轻量 √推荐 | 768 维   | 极快（⚡⚡⚡⚡⚡） | 中文专优   | 中文场景轻量首选   |

### 2.5 上下文管理：如何让 AI 记住对话？

LLM 本身无状态 —— 每次调用都是全新的。上下文管理是让它 “记住” 的关键工程

```
对话越长，需要输入的历史 Token 越多，成本呈指数级增长，同时还会触碰模型上下文窗口上限。必须用工程手段在完整性与成本之间取得平衡。
```

| 层级         | 英文名称             | 包含内容                                                   |
| ------------ | -------------------- | ---------------------------------------------------------- |
| 会话级上下文 | Session Context      | 用户身份与权限、会话开始时间、语言与风格偏好、全局配置参数 |
| 对话级上下文 | Conversation Context | 完整对话历史、当前对话主题、未解决问题堆栈、长期记忆摘要   |
| 轮次级上下文 | Turn Context         | 当前用户输入、本轮检索信息、预期输出格式、本轮工具调用     |

当对话历史过长，需要选择合适的策略进行压缩，避免 Token 爆炸

| 策略名称         | 复杂度 | 核心逻辑                                                     | 优点                               | 缺点                   |
| ---------------- | ------ | ------------------------------------------------------------ | ---------------------------------- | ---------------------- |
| 滑动窗口         | 简单   | 只保留最近 N 轮对话，超出部分直接丢弃                        | 实现极简，成本低                   | 早期关键信息易丢失     |
| 递归总结         | 中等   | 用 LLM 对旧对话生成多级摘要，逐层压缩                        | 信息保留完整                       | 生成摘要本身有额外成本 |
| 重要性采样       | 中等   | 通过关键词检测与时间衰减权重筛选重要句子                     | 能保留决策性关键信息               | 权重设计较复杂         |
| 混合策略（推荐） | 综合   | 最近 3 轮完整保留 → 3-20 轮重要性采样 → 20 轮以上递归摘要 → 超限后向量检索补充 | 兼顾回答准确与成本控制；可灵活调参 | -                      |

### 2.6 幻觉控制：如何防止 AI 编造答案？

模型可能生成 “听起来很像张三” 但实际不存在的建议或事实 —— 这叫做**幻觉（Hallucination）**。

| 策略名称         | 核心逻辑                       | 实现方式                                                     |
| ---------------- | ------------------------------ | ------------------------------------------------------------ |
| **RAG 约束接地** | 让回答必须基于真实检索到的信息 | 在 System Prompt 中明文写入：“只能基于以下事实回答”，确保所有输出都有检索结果支持，做到 “有证可查” |
| **置信度评分**   | 让 AI 诚实表达不确定性         | 对生成的信息标记置信度，低置信度内容强制加上 “我不确定” 前缀，避免把猜测伪装成事实 |
| **知识边界限制** | 阻止模型在未知领域随意发挥     | 在 System Prompt 中明确规定：“对于不在已知数据范围内的问题，回答‘我不清楚’”，强制 AI 承认自己的知识边界 |
| **关键词验证**   | 降低无根据生成的概率           | 从 RAG 检索结果中提取关键词，要求 AI 生成的答案必须包含这些关键词，确保回答和检索内容强关联 |

### 2.7 Token 成本优化：怎么让它跑得更省？

每一轮对话都需要输入完整上下文，成本会随对话轮次累积增长 —— 必须主动优化。

| 场景                   | 关键数据                                                     | 特点                 |
| ---------------------- | ------------------------------------------------------------ | -------------------- |
| **不优化时的成本增长** | 第 1 轮 ≈ 3,000 tokens第 10 轮 ≈ 15,000 tokens第 50 轮 ≈ 触碰窗口上限 | 呈线性甚至指数级增长 |
| **优化后的成本控制**   | System Message 精简 → 降低固定成本 30-50%上下文压缩 → 降低历史成本 60-80%Prompt 缓存 → 降低重复成本 高达 90% | 成本得到有效控制     |

| 优化策略                | 原理                                                   | 节省幅度 | 实现难度 |
| ----------------------- | ------------------------------------------------------ | -------- | -------- |
| System Message 精简     | 去冗余 Few-shot，压缩指令文本                          | 30-50%   | 简单     |
| 上下文压缩（混合策略）  | 滑动窗口 + 递归总结 + 向量补充                         | 60-80%   | 中等     |
